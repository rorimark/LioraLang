# Bun Code And Components Rules

## 1. Общие принципы
1. Код должен быть понятным раньше, чем умным.
2. Каждый модуль должен иметь одну четкую ответственность.
3. Читаемость, предсказуемость и безопасность важнее сокращения строк и микрооптимизаций.
4. Поведение и поддерживаемость важнее "красивых" трюков.
5. Если решение трудно быстро объяснить, его нужно упростить или жестче структурировать.

## 2. Архитектурные границы
1. Архитектура должна иметь явные слои и предсказуемое направление зависимостей.
2. Нижний слой не должен импортировать верхний.
3. UI, доменная логика, доступ к данным и runtime-специфика должны быть разделены.
4. Экранные компоненты не должны напрямую знать детали транспорта, storage, базы данных и серверной среды.
5. Если проект использует FSD-подобную схему, направление должно быть строгое: `shared -> entities -> features -> widgets -> pages -> app`.
6. Если проект использует другую структуру, правило остается тем же: зависимости направлены к более базовым слоям, а не наоборот.

## 3. Runtime и Bun-границы
1. Код должен явно разделять client-safe, server-safe и runtime-specific части.
2. Bun-специфичный код не должен протекать в браузерные компоненты и универсальную доменную логику.
3. Все обращения к файловой системе, процессам, сокетам, Bun APIs и серверной среде должны жить в отдельном инфраструктурном слое.
4. UI и shared бизнес-логика должны по возможности опираться на web standards, а не на Bun-specific API.
5. Если модуль зависит от Bun runtime, это должно быть очевидно из его расположения, имени или public API.
6. Ошибки runtime-слоя должны нормализоваться до понятных ошибок приложения до попадания в UI.

## 4. Bun как toolchain
1. Во всем проекте должен использоваться один пакетный менеджер: `bun`.
2. Источником правды для зависимостей являются `package.json` и `bun.lock`.
3. Нельзя смешивать `bun.lock` с `package-lock.json`, `pnpm-lock.yaml` или `yarn.lock`.
4. Все команды проекта должны запускаться через `bun run`.
5. Для ad-hoc утилит нужно использовать `bunx`, а не `npx`.
6. Тесты должны запускаться через `bun test`, если нет явной причины использовать другой раннер.
7. Скрипты проекта должны быть короткими, предсказуемыми и читабельными.
8. Если проект зависит от Node-compatible поведения, это должно быть проверено на совместимость с Bun заранее, а не после интеграции.

## 5. Модули и public API
1. У каждого экспортируемого модуля должен быть понятный public API.
2. Внешний код должен импортировать модуль через его публичную точку входа, а не через внутренние файлы, если это возможно.
3. Named exports являются стандартом по умолчанию.
4. Default export допустим только если это делает API реально проще, а не менее ясным.
5. Deep-import в обход структуры модуля запрещен, если модуль имеет собственный публичный вход.

## 6. Структура файлов и папок
1. Структура модулей должна быть предсказуемой и повторяемой по всему проекту.
2. Для UI-модулей стандартной структурой являются `ui`, `model`, `index`, а тесты лежат рядом с кодом.
3. `ui` содержит JSX, стили и чистое представление.
4. `model` содержит состояние, orchestration-логику, обработчики, derived values и side effects.
5. `index` экспортирует только то, что должно быть доступно снаружи.
6. Если файл становится слишком большим для быстрого сканирования, его нужно декомпозировать по ответственности.

## 7. Именование
1. Компоненты называются в `PascalCase`.
2. Хуки начинаются с `use`.
3. Утилиты, функции и переменные называются в `camelCase`.
4. Настоящие константы верхнего уровня называются в `UPPER_SNAKE_CASE`.
5. Имя должно отражать роль и смысл, а не абстрактную форму.
6. Нужно избегать расплывчатых имен вроде `data`, `item`, `thing`, `value`, если контекст можно уточнить.

## 8. Импорты
1. Импорты должны идти от внешних зависимостей к внутренним модулям.
2. Сначала библиотеки, потом shared/common модули, потом доменные слои, потом локальные файлы.
3. Внутри одного модуля допустимы относительные импорты.
4. Для межмодульных импортов лучше использовать алиасы проекта.
5. Циклические зависимости запрещены.
6. Импорт не должен раскрывать приватную структуру чужого модуля.

## 9. Общие правила написания кода
1. Код должен быть детерминированным и без скрытых сайд-эффектов.
2. Неочевидные преобразования нужно выносить в именованные функции.
3. Условные ветки должны быть короткими и легко сканируемыми.
4. Глубокую вложенность нужно уменьшать ранними `return` и декомпозицией.
5. Магические значения должны быть вынесены в именованные константы.
6. Повторяющуюся логику нужно выносить после появления устойчивого паттерна, а не преждевременно.
7. Комментарии допустимы только там, где код нельзя быстро понять из структуры.

## 10. Компоненты
1. Компонент должен быть предсказуемым, небольшим и по возможности чистым.
2. JSX должен описывать представление, а не содержать плотную бизнес-логику.
3. Подготовка данных должна происходить до `return`, а не внутри длинных JSX-выражений.
4. Если компонент становится слишком большим, его нужно делить по ответственности.
5. Презентационный компонент не должен напрямую работать с базой, файлами, runtime API и транспортом.
6. Семантические HTML-теги обязательны везде, где они отражают смысл разметки.
7. Нельзя использовать низкоуровневые конструкции вроде `createElement`, если тот же код на JSX читается проще.

## 11. Props и API компонентов
1. API компонента должно быть минимальным и понятным.
2. Если компоненту нужно слишком много пропсов, это сигнал к декомпозиции или изменению контракта.
3. Булевы пропсы должны читаться как состояние: `isLoading`, `isOpen`, `isCompact`.
4. Обработчики должны называться как действие: `onSave`, `onDelete`, `onClose`.
5. Нельзя смешивать в одном пропе несколько смыслов.
6. Лучше передать в компонент уже подготовленные данные, чем заставлять его нормализовать сырую структуру.
7. Значения по умолчанию должны быть безопасными и предсказуемыми.

## 12. Состояние и вычисления
1. В состоянии хранится только то, что действительно меняется независимо.
2. Производные данные не должны храниться как отдельный state, если их можно вычислить.
3. Дублирование состояния запрещено, если оно создает риск рассинхронизации.
4. Состояние должно жить как можно ближе к месту, где им реально управляют.
5. Локальные UI-состояния должны иметь явные и читаемые имена.
6. Сложные вычисления нужно выносить в pure-функции или usecase-модули.

## 13. Hooks и orchestration
1. Хуки должны инкапсулировать orchestration-логику, а не раздувать JSX.
2. Хук должен возвращать понятный контракт: данные, derived values, handlers и статусы.
3. Эффекты должны использоваться только для side effects.
4. Cleanup обязателен для подписок, таймеров, listeners и долгоживущих ресурсов.
5. Асинхронный код в хуках должен быть защищен от гонок и обновлений после unmount.
6. Если хук становится слишком большим, его нужно делить на несколько меньших.

## 14. Асинхронный код
1. Любая асинхронная операция должна иметь явные состояния `idle`, `loading`, `success`, `error` либо их эквиваленты.
2. UI не должен угадывать текущее состояние операции.
3. Ошибки должны нормализоваться как можно ближе к источнику.
4. Асинхронные обработчики не должны молча глотать ошибки.
5. Поведение при повторном запуске, отмене и гонках должно быть продумано заранее.
6. После успешной операции код должен явно синхронизировать локальное состояние или повторно запрашивать данные по понятному правилу.

## 15. Серверный код и доступ к данным
1. Работа с базой, файлами, очередями, кэшем и внешними API должна быть вынесена в отдельные слои.
2. Доменная логика не должна зависеть от конкретной базы данных или транспортного клиента.
3. Репозитории и сервисы должны иметь понятный контракт и стабильные ошибки.
4. Серверный код должен отдавать наружу уже нормализованные ответы.
5. Если модуль использует `Bun.file`, `Bun.write`, `Bun.serve` или другие runtime-specific API, это должно быть изолировано от клиентской части и доменной логики.

## 16. Usecases и доменная логика
1. Правила домена не должны размазываться по JSX, маршрутам, API handlers и helper-файлам.
2. Если логика используется в нескольких местах и не зависит от UI, ей не место внутри компонента.
3. Usecase-логика должна быть как можно более pure и тестируемой.
4. Оркестрация сценария должна быть отделена от форматирования UI.
5. Доменный код должен быть написан так, чтобы его можно было переиспользовать без знания конкретного экрана.

## 17. Обработка ошибок
1. Ошибки должны быть явными, нормализованными и понятными для пользователя или разработчика.
2. Технические детали не должны без фильтра уходить в UI.
3. Для recoverable-сценариев код должен возвращать понятное следующее действие.
4. Ошибка не должна оставлять приложение в противоречивом состоянии.
5. Формат ошибок должен быть консистентным по проекту.

## 18. Shared UI и общие примитивы
1. Shared UI-компоненты должны быть максимально чистыми и независимыми от бизнес-контекста.
2. Shared UI не должен импортировать экранную или фичевую логику.
3. Общие примитивы должны иметь небольшой и стабильный API.
4. Любая специфичная для конкретной страницы логика должна оставаться вне shared UI.
5. Shared UI должен быть семантичным, доступным и легко тестируемым.

## 19. Стили компонентов
1. Стили должны лежать рядом с компонентом или в централизованной дизайн-системе по понятному правилу.
2. Нужно использовать токены и шкалы, а не случайные цвета, отступы и радиусы.
3. Нельзя добавлять новые `z-index` без понимания общей шкалы слоев.
4. Inline-стили допустимы только для действительно динамических значений.
5. CSS не должен ломать семантику, доступность и взаимодействия.

## 20. Доступность
1. Интерактивный элемент должен быть интерактивным по своей семантике, а не через `div` с обработчиком.
2. Для каждого поля формы обязателен связанный `label`.
3. Иконки без текста должны иметь `aria-label`.
4. Табы, меню, тултипы, модалки и алерты должны иметь корректные роли и aria-атрибуты.
5. Компоненты должны быть управляемы с клавиатуры без нестандартных ловушек.

## 21. Тесты и bun test
1. Каждый usecase, хук или компонент с нетривиальным поведением должен иметь тесты.
2. Тест должен проверять поведение и регрессии, а не просто факт рендера.
3. Если найден баг, на него должен появиться регрессионный тест.
4. Pure-логика должна тестироваться отдельно от UI там, где это возможно.
5. Компонентные тесты должны идти через пользовательские сценарии: ввод, клик, клавиатура, состояния.
6. Тесты должны стабильно запускаться через `bun test`.
7. Моки должны быть минимальными, честными и не маскировать дефекты архитектуры.

## 22. Производительность
1. `memo`, `useMemo` и `useCallback` нельзя использовать по привычке.
2. Оптимизации добавляются только при понятной причине и реальном сигнале.
3. Не нужно создавать новые объекты, массивы и функции в горячем пути без необходимости.
4. Ключи в списках должны быть стабильными идентификаторами, а не индексами.
5. Крупные списки, фильтры и таблицы должны проектироваться с учетом стоимости рендера и объема данных.

## 23. Что запрещено
1. Нарушать направление зависимостей между слоями.
2. Смешивать в одном модуле UI, домен, runtime и инфраструктуру.
3. Протаскивать Bun-specific API в компоненты и универсальную бизнес-логику без необходимости.
4. Хранить производные данные как отдельный state без причины.
5. Строить API компонентов на случайных и плохо связанных пропсах.
6. Прятать важную логику внутри JSX.
7. Добавлять оптимизации, которые усложняют код без измеримой пользы.

## 24. Definition of Done для кода и компонентов
1. Код проходит lint, тесты и проверки архитектурных границ.
2. Нет смешения runtime-specific и client-safe кода.
3. Public API модуля понятен и не протекает внутренними деталями.
4. Компонент семантичен, доступен и не содержит лишней логики в JSX.
5. Асинхронные сценарии имеют корректные статусы, ошибки и cleanup.
6. Все новые стили используют токены и не ломают дизайн-систему проекта.
7. На изменения в нетривиальном поведении добавлены или обновлены тесты.
8. Проект последовательно использует `bun install`, `bun run`, `bunx` и `bun test`.
9. Код можно быстро прочитать, понять и безопасно изменить без скрытых ловушек.
