# Git And Commits Rules

## 1. Общие принципы
1. История Git должна помогать понимать проект, а не мешать.
2. Каждый коммит должен быть логически цельным и объяснимым сам по себе.
3. Git нужен не только для хранения кода, но и для безопасного изменения, ревью и отката.
4. Лучше сделать на один хороший коммит больше, чем один неразборчивый коммит на все сразу.
5. Чистая история ценнее, чем быстрая, но хаотичная фиксация изменений.

## 2. Основные правила работы
1. Одна задача должна жить в одной ветке.
2. Один коммит должен решать одну логическую подзадачу.
3. Не смешивать в одном коммите разные ответственности.
4. Перед каждым коммитом нужно понимать, что именно туда попадает.
5. Коммит должен быть безопасным для чтения, ревью и отката.

## 3. Ветки
1. `main` или `master` должна оставаться стабильной и пригодной к запуску.
2. Для новой задачи нужно создавать отдельную ветку.
3. Имя ветки должно быть коротким и описательным.
4. Формат именования веток должен быть консистентным, например: `feature/...`, `fix/...`, `refactor/...`, `chore/...`.
5. В одной ветке не должно быть нескольких несвязанных задач.
6. Если задача резко меняет направление, лучше создать новую ветку, чем продолжать засорять старую.

## 4. Подготовка изменений
1. Перед началом работы нужно посмотреть текущее состояние через `git status`.
2. Перед коммитом нужно просмотреть изменения через `git diff`.
3. Перед финальным коммитом нужно просмотреть staged-изменения через `git diff --cached`.
4. Нельзя коммитить файл только потому, что он уже изменился.
5. В коммит должны попадать только те изменения, которые относятся к текущей задаче.
6. Если в рабочем дереве есть чужие или несвязанные изменения, их нельзя случайно включать в коммит.

## 5. Staging
1. Stage должен быть осознанным, а не механическим.
2. Нельзя полагаться на слепой `git add .`, если есть риск захватить лишнее.
3. Если файл содержит несколько разных изменений, их нужно по возможности разделять по смыслу.
4. Перед коммитом нужно убедиться, что staged-набор соответствует сообщению коммита.
5. Если staged-изменения нельзя кратко описать одной фразой, коммит собран неправильно.

## 6. Гранулярность коммитов
1. Коммит должен быть достаточно маленьким для быстрого ревью.
2. Коммит должен быть достаточно полным, чтобы не ломать смысл изменения.
3. Логическое изменение поведения и механический рефакторинг лучше разделять.
4. Форматирование, переименование и реальное изменение логики не стоит смешивать без необходимости.
5. Отдельный багфикс, отдельный рефакторинг и отдельные тесты часто заслуживают разных коммитов.
6. Если изменение нельзя безопасно откатить отдельно, коммит, скорее всего, собран слишком широко.

## 7. Что должно быть в хорошем коммите
1. Понятная цель.
2. Понятный объем.
3. Понятный message.
4. Отсутствие несвязанных файлов.
5. Корректное состояние кода после применения коммита.
6. Желательно, чтобы проект после коммита проходил хотя бы минимальные проверки.

## 8. Чего нельзя делать
1. Нельзя коммитить "по пути" случайные правки.
2. Нельзя смешивать багфикс, новую фичу и косметику в одном коммите.
3. Нельзя оставлять в истории коммиты вида `asdf`, `tmp`, `wip`, `fix`, `changes`.
4. Нельзя коммитить сломанное состояние без явной и осознанной причины.
5. Нельзя использовать Git как мусорную корзину для промежуточных мыслей в общей истории.

## 9. Conventional Commits
1. Формат по умолчанию: `<type>(<scope>): <subject>`.
2. Если `scope` не нужен, допустим формат `<type>: <subject>`.
3. Заголовок коммита должен быть коротким, точным и читаемым без тела.
4. Сообщение должно описывать смысл изменения, а не факт редактирования файла.
5. Заголовок не должен заканчиваться точкой.

## 10. Допустимые типы коммитов
1. `feat`: новая функциональность.
2. `fix`: исправление бага.
3. `refactor`: изменение структуры кода без изменения поведения.
4. `perf`: улучшение производительности.
5. `test`: добавление или изменение тестов.
6. `docs`: документация.
7. `style`: изменения форматирования и визуальной структуры кода без изменения логики.
8. `chore`: техническое обслуживание, зависимости, конфиги, служебные задачи.
9. `build`: изменения, влияющие на сборку и toolchain.
10. `ci`: изменения пайплайнов и CI/CD.
11. `revert`: откат предыдущего коммита.

## 11. Scope
1. `scope` должен обозначать область изменения, а не техническую мелочь.
2. Хорошие scope: `auth`, `decks`, `browse`, `api`, `router`, `build`, `tests`.
3. Плохие scope: `stuff`, `misc`, `update`, `small-fix`.
4. Если изменение действительно затрагивает несколько зон, лучше разбить его на несколько коммитов.
5. `scope` должен помогать читать историю, а не просто заполнять место.

## 12. Subject
1. `subject` должен быть в повелительной или нейтральной форме и сразу объяснять результат.
2. Нужно писать, что изменилось по смыслу: `fix(auth): prevent empty password submit`.
3. Не нужно писать "edited", "updated files", "made changes".
4. Subject должен быть коротким, но не слишком общим.
5. Желательно держать заголовок коммита компактным и легко читаемым в одной строке.

## 13. Тело коммита
1. Тело нужно, когда одного заголовка недостаточно.
2. В теле нужно объяснять почему сделано изменение, а не переписывать diff.
3. Если есть важные ограничения, компромиссы или миграционные последствия, их стоит указать.
4. Если изменение потенциально рискованное, тело коммита должно это прояснять.
5. Если тело ничего не добавляет к пониманию, его можно не писать.

## 14. Breaking changes
1. Ломающие изменения должны быть помечены явно.
2. Для этого можно использовать `!` в заголовке или явное описание в теле.
3. Если после коммита нужны миграции, ручные действия или обновление API, это нужно указать.
4. Ломающие изменения нельзя прятать под обычный `fix` без пояснения.

## 15. Коммиты и тесты
1. Коммит, меняющий поведение, по возможности должен включать тест или обновление теста.
2. Исправление бага без теста допустимо только если тест действительно нельзя адекватно добавить.
3. Тесты не должны быть отдельным "хвостом", если они критичны для понимания исправления.
4. Если код и тесты слишком смешаны по смыслу, их можно разнести на два логичных коммита.

## 16. История до пуша
1. До первого пуша свою ветку можно чистить и переписывать.
2. Перед пушем история должна стать логичной и читаемой.
3. Локальные `fixup` и шумные промежуточные коммиты лучше убирать до публикации ветки.
4. Если ветка уже ушла на ревью и по ней работают другие, переписывать историю нужно очень осторожно.

## 17. Rebase, merge и обновление ветки
1. Перед merge ветку нужно синхронизировать с актуальной базой.
2. Если команда использует rebase-first workflow, нужно придерживаться его последовательно.
3. Rebase подходит для выпрямления локальной истории.
4. Merge подходит там, где важно сохранить явный факт слияния.
5. Конфликты нельзя решать механически, не понимая итоговый код.
6. После разрешения конфликтов нужно перечитать критичные файлы, а не просто завершить операцию.

## 18. Force push
1. `force push` допустим только в свою ветку и только если это действительно нужно.
2. Нельзя делать `force push` в общую стабильную ветку.
3. Перед `force push` нужно быть уверенным, что не затираются чужие изменения.
4. Если история уже обсуждается на ревью, force push нужно использовать аккуратно и осознанно.

## 19. Откат изменений
1. Для опубликованной истории предпочтителен `git revert`, а не переписывание истории.
2. Если коммит можно откатить отдельно и безопасно, это признак хорошей гранулярности.
3. Если откат одного коммита ломает все вокруг, история собрана слишком плотно.
4. Откат должен быть таким же понятным, как и исходный коммит.

## 20. Pull requests и ревью
1. Pull request должен быть посвящен одной теме.
2. История в PR должна читаться сверху вниз как последовательность понятных шагов.
3. Ревьюеру должно быть видно, где новая функциональность, где рефакторинг, где тесты.
4. Перед PR нужно самому перечитать diff как ревьюер.
5. Если PR слишком большой, его лучше разделить, даже если задача одна.

## 21. Когда коммит нужно разбить
1. Когда заголовок получается слишком общим.
2. Когда в diff смешаны несвязанные части системы.
3. Когда часть изменений можно откатить отдельно, а часть нет.
4. Когда в одном коммите одновременно есть rename, format, logic, tests и config без явной причины.
5. Когда ревью такого коммита требует читать его как хаотичный поток, а не как шаг.

## 22. Когда коммит можно объединить
1. Когда несколько мелких изменений решают одну и ту же подзадачу.
2. Когда дробление только ухудшает читаемость истории.
3. Когда код и тест составляют один атомарный смысловой шаг.
4. Когда отдельные коммиты ничего не объясняют поодиночке.

## 23. Рабочая дисциплина
1. Перед коммитом нужно знать, зачем он нужен.
2. Перед пушем нужно знать, как читается история ветки.
3. Перед merge нужно знать, что именно попадет в стабильную ветку.
4. Если есть сомнение, лучше остановиться и пересобрать историю аккуратнее.

## 24. Что считается хорошей историей
1. Ее можно читать без открытия всех файлов.
2. По сообщениям видно развитие задачи.
3. По каждому коммиту понятно, что и зачем произошло.
4. Откат и поиск регрессии не превращаются в хаос.
5. Ревьюер может двигаться по истории шаг за шагом.

## 25. Definition of Done для Git и коммитов
1. Ветка содержит одну тему и не смешивает несвязанные задачи.
2. Каждый коммит логически цельный и имеет понятный message.
3. История не содержит шумных `wip`, `tmp` и случайных фиксаций.
4. Staged-изменения проверены перед коммитом.
5. В PR нет случайных файлов, мусора и несвязанных правок.
6. Коммиты при необходимости можно безопасно откатить по отдельности.
7. История читается как последовательность осмысленных шагов, а не как поток случайных изменений.
