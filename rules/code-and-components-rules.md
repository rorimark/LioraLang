# Liora Code And Components Rules

## 1. Общие принципы
1. Код должен быть понятным раньше, чем умным.
2. Каждый модуль должен иметь одну четкую ответственность.
3. Поведение важнее сокращения строк кода.
4. Читаемость, предсказуемость и безопасность важнее микрооптимизаций.
5. Если решение сложно объяснить в двух-трех предложениях, его нужно упростить или явно структурировать.

## 2. Архитектурные слои
1. Слои проекта строго направлены снизу вверх: `shared -> entities -> features -> widgets -> pages -> app`.
2. Нижележащий слой не может импортировать вышележащий.
3. `app` отвечает за инициализацию приложения, роутинг, провайдеры и глобальные настройки.
4. `pages` собирают страницу из готовых виджетов и page-level композиции.
5. `widgets` собирают несколько сущностей и фич в законченную часть экрана.
6. `features` описывают отдельное пользовательское действие или сценарий.
7. `entities` содержат доменные представления и UI вокруг конкретной сущности.
8. `shared` содержит переиспользуемые примитивы, утилиты, конфиг, провайдеры и платформенные абстракции.
9. Бизнес-логика не должна подниматься в `shared`, если она относится к конкретной предметной области.

## 3. Платформенные границы
1. Прямой доступ к Electron, `window.electronAPI` и платформенным API запрещен вне платформенного слоя.
2. Работа с платформой идет только через `@shared/providers`, `@platform`, `@platform-target` и разрешенные репозитории.
3. Компоненты не должны знать, из web или desktop пришли данные.
4. Все различия платформ должны скрываться за адаптерами, репозиториями и gateway-абстракциями.
5. Ошибки платформенного слоя должны нормализоваться до понятных сообщений до попадания в UI.

## 4. Публичное API модулей
1. У каждого модуля должен быть понятный public API через `index.js`, если модуль экспортируется наружу.
2. Внешний код должен импортировать модуль через его публичную точку входа, а не через внутренние файлы, если это возможно.
3. Named exports являются стандартом по умолчанию.
4. Default export допустим только если это действительно оправдано и не ломает читаемость API.
5. Нельзя строить код на deep-import в обход структуры модуля.

## 5. Структура файлов и папок
1. Компонентная папка должна иметь предсказуемую структуру: `ui`, `model`, `index`.
2. `ui` содержит JSX и стили представления.
3. `model` содержит хук состояния, вычисления, события и orchestration-логику.
4. `index.js` экспортирует только то, что должно быть доступно извне.
5. Тесты должны лежать рядом с модулем, который они проверяют.
6. Если файл разрастается до уровня, где его трудно сканировать, его нужно декомпозировать, а не оставлять монолитом.

## 6. Именование
1. Имена компонентов должны быть в `PascalCase`.
2. Имена хуков должны начинаться с `use`.
3. Имена утилит и функций должны быть в `camelCase`.
4. Имена констант верхнего уровня должны быть в `UPPER_SNAKE_CASE`, если это действительно константы.
5. Имя должно описывать роль, а не форму. Лучше `handlePasswordResetRequest`, чем `doReset`.
6. Не использовать расплывчатые имена вроде `data`, `item`, `value`, если контекст можно уточнить.

## 7. Импорты
1. Импорты должны идти от внешнего к внутреннему: библиотеки, shared, entities, features, widgets, pages, локальные файлы.
2. Импорты должны оставаться стабильными и короткими через алиасы проекта.
3. Относительные импорты допустимы внутри одного модуля.
4. Если значение можно импортировать из публичного API модуля, нужно делать это через публичный API.
5. Циклические зависимости запрещены.

## 8. Общие правила написания кода
1. Код должен быть детерминированным и без скрытых сайд-эффектов.
2. Любое неочевидное преобразование данных должно иметь явное имя.
3. Условные ветки должны быть короткими и читабельными.
4. Вложенность нужно уменьшать ранними `return`, вынесением функций и декомпозицией.
5. Магические значения должны быть вынесены в именованные константы.
6. Повторяющаяся логика должна быть вынесена после появления стабильного паттерна, а не преждевременно.
7. Комментарии нужны редко и только там, где код нельзя быстро понять из структуры.

## 9. Компоненты
1. Компонент должен быть как можно более предсказуемым и по возможности чистым.
2. JSX должен описывать представление, а не содержать плотную бизнес-логику.
3. Подготовка данных должна происходить до блока `return`, а не внутри длинных JSX-выражений.
4. Если компонент становится слишком большим, его нужно делить по ответственности, а не только по размеру.
5. Презентационный компонент не должен напрямую обращаться к репозиториям и платформенным сервисам, если это можно вынести в `model`.
6. Компонент не должен знать лишнего о структуре данных, если ему можно передать уже подготовленные значения.
7. Семантические HTML-теги обязательны везде, где они отражают смысл разметки.
8. Не использовать `createElement`, если обычный JSX решает задачу проще и читабельнее.

## 10. Props и API компонентов
1. Пропсы должны образовывать понятный и минимальный API.
2. Если компоненту нужно слишком много пропсов, его нужно пересобрать или разделить.
3. Булевы пропсы должны читаться как свойства состояния: `isLoading`, `isDisabled`, `isCompact`.
4. Обработчики событий должны называться как действие: `onSave`, `onDelete`, `onOpen`.
5. Не передавать в компонент сырые структуры, если можно передать уже нормализованные значения.
6. Не смешивать в одном пропе несколько смыслов.
7. Значения по умолчанию должны быть безопасными и предсказуемыми.

## 11. Состояние и вычисления
1. В `state` хранится только то, что действительно меняется независимо.
2. Производные данные не должны храниться в `state`, если их можно вычислить из текущего состояния и пропсов.
3. Дублирование состояния запрещено, если это создает риск рассинхронизации.
4. Состояние должно жить как можно ближе к месту, где им реально управляют.
5. Временные UI-состояния должны быть явно названы: `isOpen`, `activeTab`, `pendingAction`, `errorText`.
6. Сложные преобразования данных должны быть вынесены в отдельные функции или usecase-слой.

## 12. Hooks и model-слой
1. Хуки `model` являются основной точкой orchestration-логики для виджетов и фич.
2. Хук должен возвращать понятный контракт: данные, derived values, handlers и статусы.
3. Эффекты должны использоваться только для сайд-эффектов, а не для вычисления обычных значений.
4. Подписки, таймеры и внешние ресурсы обязаны иметь cleanup.
5. Асинхронный код в хуках должен защищаться от гонок и обновлений после unmount.
6. Обработчики событий и эффекты должны быть логически разделены.
7. Если хук становится слишком большим, его нужно декомпозировать по ответственности.

## 13. Асинхронный код и данные
1. Асинхронные операции должны иметь явные состояния `idle`, `loading`, `success`, `error` либо их эквиваленты.
2. UI не должен угадывать, что происходит. Компонент должен получать явный статус.
3. Ошибки должны нормализоваться как можно ближе к источнику.
4. После успешной операции код должен явно обновлять локальное состояние или повторно запрашивать данные по понятному правилу.
5. Асинхронные обработчики не должны молча проглатывать ошибки.
6. Любой запрос, влияющий на интерфейс, должен иметь понятное поведение при повторном запуске, отмене и гонках.

## 14. Репозитории, usecases и доменная логика
1. Репозитории отвечают за доступ к данным и внешним системам.
2. UI не должен знать детали транспорта, storage или платформенной реализации.
3. Usecase-логика выносится туда, где ее можно тестировать отдельно от React.
4. Правила домена нельзя размазывать по JSX, стилям и случайным helper-файлам.
5. Если логика используется в нескольких местах и не зависит от UI, ей не место внутри компонента.

## 15. Обработка ошибок
1. Ошибки должны быть явными, локализуемыми и понятными для пользователя.
2. Техническая ошибка не должна без фильтра уходить в интерфейс.
3. Для recoverable-сценариев UI должен предлагать следующее действие.
4. Логику отображения ошибок нужно держать консистентной по проекту.
5. Ошибка не должна оставлять интерфейс в противоречивом состоянии.

## 16. Компоненты shared/ui
1. Компоненты в `packages/shared/src/ui` являются примитивами и не должны зависеть от бизнес-контекста приложения.
2. Shared UI-компонент не должен импортировать код из `features`, `widgets`, `pages` или `app`.
3. Такие компоненты должны иметь максимально чистый, стабильный и небольшой API.
4. Любая специфичная для экрана логика должна оставаться вне shared UI.
5. Shared UI должен быть семантичным, доступным и переиспользуемым без знания конкретной страницы.

## 17. Стили компонентов
1. Стили компонента должны лежать рядом с компонентом.
2. Для классов нужно использовать предсказуемую схему именования блока, элементов и модификаторов.
3. Стили должны использовать токены, а не случайные значения.
4. Нельзя добавлять новые `z-index` без явной необходимости и понимания существующей шкалы слоев.
5. Inline-стили допустимы только для реально динамических значений, которые неудобно или невозможно выразить через классы.
6. CSS не должен ломать семантику и взаимодействия компонента.

## 18. Доступность на уровне кода компонентов
1. Интерактивные элементы должны быть интерактивными по своей семантике, а не через `div` с обработчиками.
2. Для полей формы обязателен связанный `label`.
3. Иконки без текстовой подписи должны иметь `aria-label`.
4. Табы, меню, тултипы, модалки и алерты должны иметь корректные роли и aria-атрибуты.
5. Компонент должен быть управляем с клавиатуры без нестандартных ловушек.

## 19. Тесты
1. Каждый usecase, хук или компонент с нетривиальным поведением должен иметь тесты.
2. Тест должен проверять поведение и регрессии, а не просто факт рендера.
3. Если был найден баг, на него должен появиться регрессионный тест.
4. Pure-логика тестируется отдельно от React там, где это возможно.
5. Компонентные тесты должны идти через поведение пользователя: ввод, клик, клавиатура, состояния.
6. Тесты не должны быть написаны так, чтобы просто проходить при любой реализации.
7. Моки должны быть минимальными и честными.

## 20. Производительность
1. `memo`, `useMemo` и `useCallback` нельзя использовать по привычке.
2. Оптимизации добавляются только при понятной причине: стабильный API, дорогие вычисления, чувствительный список, реальный ререндер-проблема.
3. Не создавать новые массивы, объекты и функции в горячем пути без необходимости.
4. Ключи в списках должны быть стабильными идентификаторами, а не индексами.
5. Большие списки и таблицы должны строиться с учетом пагинации, фильтрации и стоимости рендера.

## 21. Что запрещено
1. Нарушать направление слоев.
2. Импортировать Electron и платформенные детали напрямую из UI.
3. Хранить производные данные как отдельный state без необходимости.
4. Строить API компонента на случайных пропах без структуры.
5. Прятать важную бизнес-логику в JSX.
6. Добавлять оптимизации, которые усложняют код без измеримой пользы.
7. Смешивать в одном модуле платформенную, доменную и презентационную ответственность.

## 22. Definition of Done для кода и компонентов
1. Код проходит lint, тесты и проверки границ слоев.
2. Нет нарушений архитектуры и прямых платформенных утечек.
3. Public API модуля понятен и не протекает внутренними деталями.
4. Компонент семантичен, доступен и не содержит лишней логики в JSX.
5. Асинхронные сценарии имеют корректные состояния, ошибки и cleanup.
6. Все новые стили лежат рядом с компонентом и используют токены.
7. На изменения в нетривиальном поведении добавлены или обновлены тесты.
8. Код можно быстро прочитать, понять и безопасно изменить без скрытых ловушек.
