# Liora UI Rules

## 1. Продуктовое ощущение
1. Интерфейс должен быть спокойным: контент важнее декора.
2. Каждый экран сразу отвечает на 3 вопроса: где я, что могу сделать, что сейчас важно.
3. Один экран должен иметь одну главную задачу, остальные действия вторичны.
4. Нельзя использовать визуальный шум: без градиентов, без лишнего свечения, без тяжелых декоративных эффектов.
5. Плотность интерфейса должна быть сбалансированной: не пусто и не перегружено.

## 2. Визуальный язык
1. Базой являются нейтральные поверхности, аккуратные границы и сдержанные акценты.
2. Тени должны быть мягкими и короткими, только для разделения слоев.
3. Радиусы должны быть умеренными и едиными для всех компонентов.
4. Акцентный цвет должен быть один и использоваться последовательно.
5. Контраст текста должен быть выше контраста декоративных элементов.

## 3. Токены и стили
1. Нельзя использовать raw-цвета в компонентах, только токены.
2. Нужно разделять базовые токены и семантические токены.
3. Размеры, отступы, радиусы, тени и z-index задаются только через шкалы токенов.
4. Цветовые токены должны поддерживать тему light, dark и system.
5. Имена токенов должны быть стабильными и предсказуемыми.

## 4. Типографика
1. Для интерфейсного текста используется одна основная гарнитура.
2. Шкала размеров и line-height должна быть фиксированной, без случайных значений.
3. Заголовки должны быть краткими и визуально плотными.
4. Основной текст должен быть легко читаемым и менее плотным, чем заголовки.
5. Подписи и мета-тексты всегда менее контрастные, чем основной текст.
6. Ширина строк должна оставаться комфортной для чтения.

## 5. Сетка и отступы
1. Нужно использовать единую шкалу отступов на всем проекте.
2. Компоненты должны быть выровнены по единому вертикальному ритму.
3. Контент не должен прилипать к краям контейнеров.
4. Внутренние отступы панелей, карточек и форм должны быть консистентными.
5. На мобильных обязательно учитывать safe-area.

## 6. Навигация
1. Навигация должна быть предсказуемой и одинаковой по паттернам на всех страницах.
2. Активный раздел должен быть визуально очевиден в любой момент.
3. Клик по активной вкладке не должен вызывать лишний ререндер и перезагрузку данных.
4. Мобильная навигация должна работать плавно без задержек и ложных состояний.
5. Иконки и текст в навигации должны быть строго выровнены.

## 7. Кнопки и действия
1. В каждом блоке должна быть только одна primary-кнопка.
2. Secondary и ghost действия должны быть заметно тише, чем primary.
3. Для всех кнопок обязательны состояния default, hover, active, focus, disabled, loading.
4. На мобильных ключевые действия должны быть удобными для большого пальца.
5. Иконка и текст в кнопках должны быть выровнены по центру по вертикали.
6. Для опасных операций нужен явный confirm.

## 8. Формы
1. Обязательно использовать семантические теги form, label и button type="submit".
2. Enter должен отправлять форму в ожидаемом контексте.
3. Label обязателен для каждого поля, placeholder не заменяет label.
4. Ошибки валидации должны показываться рядом с полем и понятным языком.
5. Тексты в чувствительных разделах должны объяснять текущее состояние пользователя.
6. Нужно использовать корректные значения autocomplete.

## 9. Таблицы, списки и карточки
1. Заголовки колонок и действия должны быть строго выровнены.
2. Клик-зона строки должна быть предсказуемой и одинаковой для всех строк.
3. Меню действий через многоточие используется только когда это улучшает сканирование.
4. Поиск, фильтры и счетчики должны быть компактными и читаемыми.
5. Тултипы и меню не должны обрезаться контейнерами.
6. Пагинация должна быть понятной и доступной с клавиатуры.

## 10. Тултипы, поповеры и модалки
1. Нельзя использовать системный title для важных подсказок.
2. Кастомный тултип должен иметь корректный слой и не перекрываться таблицей.
3. Стрелка тултипа должна быть геометрически привязана к триггеру.
4. Модалка должна иметь фокус-трап и корректное закрытие по Escape, если это безопасно.
5. Оверлеи не должны мерцать или прыгать при скролле.

## 11. Состояния и обратная связь
1. Для каждого ключевого сценария обязательны состояния loading, empty, error, success.
2. Ошибки должны объяснять что произошло, почему и что делать дальше.
3. Статусы не должны исчезать слишком рано.
4. Локальные ошибки показываются рядом с контекстом, а не только глобально.
5. После важного действия пользователь должен видеть явный результат.

## 12. Анимации и производительность
1. Анимации должны быть короткими и функциональными.
2. Нельзя использовать анимации, которые блокируют ввод.
3. Предпочтение нужно отдавать transform и opacity.
4. Для prefers-reduced-motion анимации должны упрощаться.
5. Быстрые повторные взаимодействия не должны вызывать лаги.

## 13. Адаптив и мобильный UX
1. Интерфейс должен корректно работать в portrait и landscape.
2. Нужно учитывать safe-area со всех сторон, включая устройства с вырезами.
3. Нельзя допускать случайный zoom при вводе и при смене ориентации.
4. Минимальный touch target для интерактивных элементов должен быть 44x44.
5. В мобильном режиме ключевые действия должны быть в приоритете.

## 14. Доступность
1. Контраст текста и интерактивов должен соответствовать WCAG.
2. Видимый focus-ring обязателен для клавиатурной навигации.
3. Для tabs, menu, tooltip и modal обязательны корректные ARIA-атрибуты.
4. Tab-order должен быть логичным и без ловушек.
5. Иконки без текста должны иметь aria-label.

## 15. Тексты и microcopy
1. Тексты должны быть короткими, конкретными и без лишнего технического жаргона.
2. Кнопки должны называться действием, а не абстрактными словами.
3. Формулировки не должны пугать пользователя без причины.
4. Текст должен снижать неопределенность и помогать принять следующее действие.

## 16. Definition of Done для UI-задачи
1. Изменение не ломает существующие сценарии.
2. Результат консистентен со всем интерфейсом Liora.
3. Проверено на desktop, mobile и landscape.
4. Проверена работа с клавиатуры и базовая доступность.
5. Проверены состояния loading, empty, error, success.
6. Формы корректно отправляются по Enter и валидируются до отправки.
7. Нет лишних ререндеров при повторных кликах по активным разделам.
8. Новые стили выполнены через токены и не нарушают визуальную систему.
