# Project Overview

## Продукт

LioraLang - это локальное приложение для изучения слов и выражений через собственные колоды.

В центре продукта четыре сценария:

1. создать или импортировать колоду;
2. наполнить ее словами, тегами, примерами и языковыми метаданными;
3. учить карточки в режиме `Learn` по SRS-правилам;
4. анализировать прогресс и при необходимости делиться колодой через Hub.

## Ключевые пользовательские возможности

### Колоды

- создание, редактирование, переименование и удаление;
- импорт из JSON и `lioradeck`;
- экспорт в JSON и `lioradeck`;
- просмотр деталей колоды и списка слов;
- базовые теги, языки и примеры использования.

### Изучение

- SRS-сессии с оценками `Again`, `Hard`, `Good`, `Easy`;
- учёт дневных лимитов;
- дополнительная сессия после исчерпания лимитов;
- клавиатурные шорткаты;
- сохранение части состояния между переходами.

### Прогресс

- недельная активность;
- интенсивность по дням;
- статистика по колодам;
- split по retention;
- базовые milestones.

### Настройки

- тема интерфейса;
- параметры SRS и study session;
- настройки импорт/экспорт;
- shortcuts;
- desktop-настройки, integrity и обновления.

### Hub

- просмотр публичных колод;
- импорт опубликованных пакетов;
- публикация локальной колоды;
- учёт скачиваний.

## Web и Desktop

### Desktop

- основная офлайн-версия;
- хранит данные в SQLite;
- работает через Electron main/preload;
- умеет менять путь к БД, проверять целостность и скачивать обновления.

### Web

- браузерная и PWA-версия;
- хранит данные в IndexedDB;
- использует service worker и manifest;
- часть desktop-функций заменяет заглушками.

## Сильные стороны проекта

- сильный offline-first фокус;
- понятная доменная модель: deck, word, review card, settings;
- единый React-рендерер для двух платформ;
- хорошая база для дальнейшего роста: Hub, PWA, desktop shell, import/export;
- достаточно аккуратная разметка страниц и единый UI-слой в `shared/ui`.

## Слабые стороны проекта

- архитектура FSD соблюдается непоследовательно;
- несколько критически важных файлов разрослись слишком сильно;
- часть кода дублируется;
- есть мертвые или недоделанные абстракции;
- account-слой пока только заглушка;
- документация и часть README были не синхронизированы со скриптами проекта.

## Куда проекту логично расти дальше

- полноценный account/sync слой;
- тесты для SRS, import/export и платформенных адаптеров;
- строгий контроль FSD-границ на уровне lint/CI;
- разбиение крупных модулей на более простые use case-ы и подхуки;
- усиление security-политики desktop-версии;
- улучшение наблюдаемости: error reporting, perf markers, bundle analysis.
